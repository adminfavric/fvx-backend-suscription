# Internacionalización (i18n) — backend

Complementa la guía del cliente en [`../../fvx-frontend/docs/i18n.md`](../../fvx-frontend/docs/i18n.md): allí se describe Transloco, JSON y el interceptor **`Accept-Language`**; aquí se centra en **Django**.

## Configuración

En `core/settings.py`:

- **`USE_I18N`**, **`LANGUAGES`** (`es`, `en`), **`LOCALE_PATHS`** (catálogos en `fvx-backend/locale/`).
- **`LocaleMiddleware`** en `MIDDLEWARE` (debe ir después de `SessionMiddleware` y antes de `CommonMiddleware`, según la documentación de Django).

El idioma efectivo de cada petición puede venir de **`Accept-Language`**, sesión o `LANGUAGE_CODE` por defecto (también configurable por entorno).

## Choices y serializers

- **`api/choices.py`**: etiquetas humanas con **`gettext_lazy`**; los **códigos** (`ADMIN`, `EDITOR`, …) permanecen estables para el JSON.
- **`api/serializers/`** (p. ej. `user.py`): en representaciones de perfil se expone **`role_label`** (y en cada organización del usuario, **`role_label`** junto a **`role`**) resolviendo la traducción para el locale de la petición.

El front no debe depender del texto de `role_label` como clave de lógica; solo para mostrar al usuario. La lógica debe usar **`role`** (código).

## Catálogos gettext

Tras modificar cadenas envueltas en `_()` / `gettext_lazy` en el código Python, actualiza y compila los catálogos **dentro del contenedor** del servicio `web` (el proyecto está montado en `/app`; los `.po` editables quedan en tu host bajo `fvx-backend/locale/`).

```bash
# Desde el directorio fvx-backend/ (donde está docker-compose.yml), con el stack levantado
docker compose exec web python manage.py makemessages -l en -l es
# Editar en el host: locale/*/LC_MESSAGES/django.po
docker compose exec web python manage.py compilemessages
```

Más contexto de arranque con Docker: [`../README.md`](../README.md).

## Alineación con el frontend

- Idioma por defecto de la **UI** Angular: configurado en `app.config.ts` (Transloco; p. ej. `en`).
- Idioma por defecto de **Django**: `LANGUAGE_CODE` en settings (puede ser `es` u otro según `.env`).

No tienen por qué coincidir; sí debe quedar claro en producto qué espera el usuario el primer día. El front fuerza el header **`Accept-Language`** según el idioma que elija en la app, de modo que las etiquetas del API (p. ej. `role_label`) coincidan con ese idioma aunque el `LANGUAGE_CODE` del servidor sea otro.

## Resumen: qué ejecutar con Docker

En **`fvx-backend/`** (junto a `docker-compose.yml`), con contenedores en marcha (por ejemplo `docker compose up -d` o el flujo que uses del [`README.md`](../README.md)):

```bash
docker compose exec web python manage.py makemessages -l en -l es
docker compose exec web python manage.py compilemessages
```

1. Primera línea: regenera o crea entradas en `locale/*/LC_MESSAGES/django.po`.  
2. Edita los `.po` en el repo (volumen montado).  
3. Segunda línea: genera los `.mo` que Django usa en runtime.
